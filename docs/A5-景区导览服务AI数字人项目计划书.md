# A5 景区导览服务 AI 数字人项目计划书

**项目名称：** 灵境导游 - 景区导览服务 AI 数字人系统

**参赛赛题：** 第十五届中国软件杯 A5 - 景区导览服务AI数字人

**出题企业：** 锐捷网络（苏州）有限公司

**适用阶段：** 初赛立项、产品方案设计、系统开发排期、答辩材料准备

---

## 1. 项目概述

### 1.1 项目背景

文旅产业正在从传统人工讲解、固定录音导览、静态图文展示，转向智能化、互动化和个性化服务。景区在旺季常面临导游资源不足、服务质量不稳定、游客问题无法实时解答、反馈数据难以沉淀等问题。传统导览设备通常只能单向播放固定内容，无法根据游客兴趣、当前位置、游览时长和情绪反馈动态调整讲解策略。

本项目面向 A5 赛题要求，建设一个具备多模态交互能力的景区导览服务 AI 数字人系统。系统以“AI 数字人导游”为核心入口，结合本地景区知识库、语音识别、语音合成、大模型问答、路线推荐、游客行为分析和管理后台，形成游客服务与景区运营双侧闭环。

### 1.2 项目目标

项目目标是开发一个完整、稳定、可演示、可扩展的智能导览系统，覆盖游客端和管理端两个核心业务面。

游客端目标：

1. 支持文本和语音提问，游客可以像和真人导游交流一样询问景区历史、景点特色、开放时间、游览路线、拍照点位等问题。
2. 数字人以语音、字幕、口型同步和表情动作进行回答，形成比普通聊天机器人更强的沉浸感。
3. 基于游客兴趣、同行人群、游览时长和偏好，推荐差异化路线和讲解重点。
4. 支持景点讲解、路线导航、问答追问、游客反馈采集，形成完整游览流程。

管理端目标：

1. 管理员可以维护景区知识库，包括景点资料、讲解词、FAQ、开放时间、路线信息和活动公告。
2. 管理员可以配置数字人的形象、服装、声音、人设风格和讲解语气。
3. 系统可以分析游客交互记录和行为数据，输出关注点分析、满意度趋势、情感趋势和运营建议。
4. 数据大屏展示服务人次、热门问题、热门景点、平均响应时长、满意度变化、消费和停留行为等指标。

### 1.3 项目定位

本项目不只是“数字人外壳 + 聊天机器人”，而是一个面向智慧景区运营的 AI 导览产品。核心竞争力体现在三个方面：

1. 准确可信：基于官方资料构建本地知识库，用 RAG 检索增强生成降低幻觉，确保景区事实问答准确率达到比赛要求。
2. 自然互动：通过语音识别、语音合成、字幕、口型驱动和表情状态，让 AI 导游具备接近真人讲解的表达能力。
3. 运营闭环：把游客交互和行为数据沉淀为管理后台洞察，帮助景区优化路线、讲解内容、服务资源和营销策略。

## 2. 赛题要求解读与评分策略

### 2.1 官方核心要求

赛题要求开发一个具备多模态交互能力的 AI 数字人导游软件，包含游客交互端和管理后台。系统需要实现实时智能问答、个性化路线讲解、情感互动，并为景区管理方提供游客反馈分析。

官方基本功能可以拆解为七类能力：

1. 语音输入和文本输入。
2. 数字人语音回答、表情响应和口型同步。
3. 景区历史、文化、景点特色等问答。
4. 根据游客兴趣推荐游览路线和讲解重点。
5. 景区知识库上传、更新和维护。
6. 数字人外观、服装、声音等配置。
7. 游客感受度报告和数据大屏概览。

非功能要求包括：

1. 至少使用 1 个多模态大模型作为核心 AI 能力支撑。
2. 构建本地景区知识库。
3. 景区事实性问答准确率不低于 90%。
4. 语音问答延迟控制在合理范围，建议小于 5 秒。
5. 系统稳定，不出现崩溃或长时间无响应。
6. 鼓励考虑 GPS 信号弱、定位困难等复杂现实场景。

### 2.2 评分项对齐

| 评分项 | 分值 | 对应建设内容 | 答辩呈现重点 |
| --- | ---: | --- | --- |

| 技术与创新性 | 30 | 多模态大模型、RAG 知识库、口型同步、情感分析、路线推荐 | 展示系统架构图、检索证据、模型调用链、准确率测试 |

| 文档质量 | 10 | 总体设计文档、部署手册、PPT、演示视频、测试报告 | 材料结构清晰、数据充分、亮点突出 |

### 2.3 竞赛取胜策略

本项目优先确保“完整度”和“可演示性”，在此基础上突出 AI 技术深度和行业实用价值。

第一优先级：做出完整闭环。游客从进入系统、选择身份偏好、语音提问、收到数字人讲解、获得路线推荐、提交反馈；管理员从上传知识、查看问答记录、查看热点统计、生成报告，必须可以完整跑通。

第二优先级：保证问答准确。评审会重点看景区事实问答，系统必须能够展示“答案来自知识库证据”，减少大模型编造。

第三优先级：数字人表现力。口型、表情、声音、字幕和动作要和回答内容联动，哪怕不是重型 3D，也要做到自然、稳定、可控。

第四优先级：数据价值。后台不能只是普通表格，要能把游客行为和满意度变成景区管理建议，例如“历史文化类问题增长，建议加强灵山大佛和梵宫讲解内容推荐”。

## 3. 用户与业务场景

### 3.1 目标用户

游客用户：

1. 首次到访景区的普通游客，需要快速了解景点、路线和注意事项。
2. 亲子游客，希望获得轻松、生动、适合儿童理解的讲解。
3. 历史文化爱好者，希望获得深度文化背景和典故解释。
4. 老年游客，希望获得语音交互、慢节奏路线和休息点推荐。
5. 时间有限游客，希望系统推荐 1-2 小时精华路线。

景区管理者：

1. 运营人员，需要查看游客关注点、热门问题、满意度和服务压力。
2. 内容管理员，需要维护景点资料、讲解词和活动公告。
3. 服务管理人员，需要分析游客反馈，优化服务资源配置。
4. 品牌宣传人员，需要通过数字人形象强化景区文化特色。

### 3.2 核心业务场景

场景一：游客问答

游客问：“灵山大佛为什么高 88 米？”系统识别问题后，从本地知识库检索灵山大佛相关资料，生成简洁准确的回答，并通过数字人语音讲解，同时在界面展示参考景点卡片。

场景二：个性化路线推荐

游客选择“我喜欢历史文化，有 4 小时时间，同行有老人”，系统推荐文化深度但步行压力适中的路线，并说明每个景点的讲解重点、预计停留时间和休息点。

场景三：动态讲解

游客点击“九龙灌浴”，数字人自动切换到讲解模式，介绍景点背景、演出时间、最佳观赏位置和拍照建议。

场景四：情感互动

游客反馈“排队太久了，有点烦”，系统识别负面情绪，数字人先安抚，再推荐附近低拥挤景点、休息点或替代路线。

场景五：管理后台分析

管理员查看当日数据大屏，发现“演出时间”“路线推荐”“灵山大佛历史”是高频问题，满意度在下午时段下降。系统生成建议：在游客端首页增加演出时间快捷入口，下午推送错峰路线。

## 4. 产品方案设计

### 4.1 产品整体结构

系统由五个子系统组成：

1. 游客交互端：面向游客的 Web/H5 或 App 界面，承载数字人、语音问答、路线推荐和景点讲解。
2. AI 服务层：负责语音识别、大模型问答、知识库检索、路线推荐、情感分析和语音合成。
3. 数字人渲染层：负责数字人形象展示、口型同步、表情动作、字幕和讲解状态。
4. 管理后台：负责知识库管理、数字人配置、交互记录、数据报表和游客感受度报告。
5. 数据与知识层：存储景点文档、向量索引、游客行为数据、问答日志、满意度和系统配置。

### 4.2 游客端功能

#### 4.2.1 首页与游客画像

游客进入系统后，首页展示数字人导游、语音按钮、文本输入框、热门问题、推荐路线和景点入口。首次使用时，系统引导游客选择兴趣和游览约束：

1. 兴趣类型：历史文化、自然风光、亲子互动、拍照打卡、休闲祈福、深度研学。
2. 游览时长：1 小时、2 小时、半日、全天。
3. 同行人群：独自、情侣、朋友、亲子、老人同行、团队。
4. 体力偏好：轻松、适中、深度。

这些信息用于路线推荐、讲解风格和回答内容长度控制。

#### 4.2.2 多模态问答

游客可以通过语音或文本提问。语音输入经过 ASR 转写为文本，系统将问题、游客画像、当前景点、历史对话和检索到的知识片段一起输入大模型，生成答案。

回答展示包括：

1. 数字人语音播报。
2. 文字字幕同步显示。
3. 景点卡片或路线卡片。
4. 相关追问建议，例如“这里适合拍照吗”“附近有什么休息点”“需要多久游览”。

#### 4.2.3 景点讲解

景点讲解采用“短讲解 + 深入追问”的模式。系统默认输出 60-90 秒内可听完的讲解，避免游客在移动场景中负担过重。游客可以点击“讲深一点”“儿童版讲解”“拍照建议”“历史典故”等按钮切换讲解角度。

讲解内容来源于官方资料包中的结构化景点数据和文化指南文档。每个景点建立独立知识条目，包括景点名称、位置、景观特色、文化内涵、详细介绍、游览亮点、开放或演出信息、备注。

#### 4.2.4 个性化路线推荐

路线推荐综合考虑游客兴趣、游览时长、同行人群、体力偏好、景点热度和预计停留时间。

预设路线包括：

1. 历史文化深度路线：适合文化爱好者，突出灵山胜境历史、佛教文化和核心建筑。
2. 轻松休闲路线：适合老人和亲子，减少长距离步行，优先安排休息点和高价值景点。
3. 精华打卡路线：适合时间有限游客，覆盖标志性景点和拍照点。
4. 自然风光路线：突出太湖、山体、园林和景观视角。
5. 研学讲解路线：面向学生团队，强调历史脉络、文化知识和互动问答。

推荐结果展示路线图、景点顺序、预计耗时、讲解重点、注意事项和替代方案。

#### 4.2.5 情感互动与反馈

系统在对话中识别游客情绪倾向，包括满意、疑惑、焦急、抱怨、疲惫等状态。游客端提供简短反馈入口，支持评分、标签和语音反馈。

当检测到负面情绪时，数字人回答策略调整为：

1. 先表达理解和安抚。
2. 给出具体解决建议。
3. 推荐附近替代景点、休息点或错峰路线。
4. 将问题归类进入后台感受度报告。

### 4.3 管理后台功能

#### 4.3.1 数据大屏

数据大屏展示核心运营指标：

1. 今日服务人次、本周服务人次。
2. 问答总量、语音交互占比、平均响应时长。
3. 热门景点排行、热门问题排行。
4. 游客满意度趋势、情感趋势。
5. 游客停留时长、消费结构、同行人数分布。
6. 知识命中率、无答案问题数量。

#### 4.3.2 知识库管理

管理员可以上传 Word、Excel、PDF、Markdown、TXT 等资料。系统将文档解析、切分、清洗、向量化并进入知识库。每条知识支持分类、标签、版本、启停状态和更新时间。

知识库管理页面提供：

1. 文档上传。
2. 文档解析状态。
3. 知识片段检索。
4. 高频未命中问题。
5. 问答测试入口。
6. 知识版本回滚。

#### 4.3.3 数字人形象管理

数字人配置包括：

1. 形象：文化导游、亲子导游、研学讲师等风格。
2. 服装：现代导游服、景区文化服饰、节庆服饰。
3. 声音：普通话女声、普通话男声、温柔讲解、活泼亲子。
4. 表情策略：微笑、思考、惊喜、安抚、讲解强调。
5. 人设语气：专业、亲切、活泼、沉稳。

比赛阶段可以优先实现 2D 数字人或 Live2D 风格形象，保证稳定和效果。若时间允许，再扩展 3D 模型或更复杂动作。

#### 4.3.4 游客感受度报告

系统按日、周、月生成游客感受度报告，内容包括：

1. 游客关注点：哪些景点、问题、路线最受关注。
2. 情感趋势：正向、中性、负向反馈比例及变化。
3. 满意度趋势：按时间、景点、游客类型分析。
4. 高频问题：游客最常问的问题和当前知识覆盖情况。
5. 服务建议：系统自动生成可执行运营建议。

报告示例：

“本周游客对九龙灌浴演出时间关注度最高，占全部问题 18.6%。下午 14:00-16:00 负向情绪比例上升，主要原因集中在排队、路线不清晰和休息点咨询。建议在游客端首页增加演出时间卡片，并在下午时段主动推荐错峰路线。”

## 5. 技术方案

### 5.1 推荐技术栈

前端：

1. React 或 Vue 作为游客端和后台基础框架。
2. Tailwind CSS 或 Ant Design Vue/React 用于后台管理界面。
3. Web Audio API 实现录音、播放和语音状态控制。
4. Canvas、Live2D、VRM 或轻量数字人 SDK 实现数字人展示。

后端：

1. Python FastAPI 或 Node.js NestJS 提供 API 服务。
2. PostgreSQL 或 MySQL 存储用户、日志、配置和业务数据。
3. Milvus、Qdrant、Chroma 或 FAISS 存储向量索引。
4. Redis 用于会话缓存、热点问题缓存和任务状态。

AI 能力：

1. 大模型：使用可调用 API 的多模态大模型或开源大模型。
2. Embedding：使用中文表现稳定的文本向量模型。
3. ASR：Whisper、FunASR 或云服务语音识别。
4. TTS：Edge TTS、CosyVoice、ParaTTS 或云服务语音合成。
5. 情感分析：大模型分类或轻量中文情绪分类模型。
6. 口型同步：基于 TTS 音频生成 viseme 序列，驱动 2D/3D 数字人口型。

### 5.2 AI 问答流程

完整问答链路如下：

1. 接收游客语音或文本输入。
2. 语音输入先经过 ASR 转写。
3. 对问题进行意图识别，判断是景点问答、路线推荐、闲聊、投诉反馈还是操作指令。
4. 对景区问题进行知识库检索，召回相关景点资料、讲解词和 FAQ。
5. 对召回结果进行重排序，筛选最相关的 3-5 条知识片段。
6. 将游客问题、游客画像、当前景点、历史对话和知识片段交给大模型生成回答。
7. 对回答进行事实约束检查，确保不回答知识库没有支持的具体事实。
8. 生成字幕、语音和数字人动作。
9. 存储问题、答案、命中知识、耗时、满意度和情绪结果。

### 5.3 RAG 知识库方案

知识库建设是本项目准确率的核心。

数据来源：

1. 官方示范景区公开资料包中的结构化景点数据。
2. 官方示范景区历史、文化、景点特色与个性化游览指南。
3. 景点行为分析 Excel 中的景区和游客行为字段。
4. 后台管理员后续上传的讲解词、FAQ、活动公告。

知识切分策略：

1. 按景点名称建立一级分组。
2. 按“位置、特色、文化内涵、详细介绍、游览亮点、开放信息、路线建议”建立二级标签。
3. 每个知识片段控制在 300-600 中文字，保留景点名称和字段标签。
4. 对演出时间、开放时间、票价、注意事项等事实类信息建立结构化字段。

检索策略：

1. 使用向量检索召回语义相关内容。
2. 使用关键词检索补充景点名、演出时间、地点等精确匹配。
3. 使用混合检索得分排序。
4. 对相同景点的多个片段进行合并。
5. 答案中优先使用结构化字段，长文本用于文化讲解。

防幻觉策略：

1. 提示词要求模型只能基于知识库回答景区事实。
2. 对没有知识支持的问题，回答“当前资料中未提供该信息”，并建议咨询工作人员。
3. 管理后台记录无答案问题，供管理员补充知识。
4. 测试集中覆盖事实问答、路线推荐、模糊问法和干扰问题。

### 5.4 路线推荐方案

路线推荐采用“规则约束 + 大模型解释”的方式，确保结果稳定可控。

规则层负责：

1. 根据游览时长限制景点数量。
2. 根据同行人群调整步行强度。
3. 根据兴趣类型选择景点标签。
4. 根据演出时间安排动态景点。
5. 根据景点预计停留时间计算路线总时长。

大模型负责：

1. 将路线解释成自然语言。
2. 生成不同游客类型的讲解重点。
3. 给出替代路线和注意事项。
4. 根据游客追问微调推荐理由。

### 5.5 数字人方案

数字人方案以“稳定可演示”为优先级。

基础版：

1. 使用 2D 数字人形象。
2. 根据 TTS 音频驱动口型开合。
3. 根据回答类型切换表情：讲解、思考、微笑、安抚、提醒。
4. 播放语音时同步字幕。

增强版：

1. 引入 Live2D 或 VRM 模型。
2. 支持头部轻微动作、眨眼、手势和情绪动作。
3. 支持不同服装和声音配置。
4. 支持景区文化主题形象。

答辩演示重点不是模型复杂度，而是数字人的“回答内容、语音、口型、表情、字幕”能形成一致体验。

### 5.6 数据分析方案

Excel 行为分析数据可用于模拟景区运营数据。核心字段包括游客 ID、昵称、年龄、性别、景点名称、景点类型、游览日期、停留时长、门票成本、餐饮消费、购物消费、交通消费、娱乐消费、总消费、同行人数和满意度。

后台分析指标：

1. 游客画像：年龄、性别、同行人数分布。
2. 景点热度：访问次数、平均停留时长、满意度。
3. 消费结构：餐饮、购物、交通、娱乐、人均消费。
4. 满意度分析：按景点、日期、游客类型分组。
5. 运营建议：高热低满意景点需要优化服务，高停留高消费景点适合重点推荐。

## 8. 关键创新点

### 8.1 可解释 RAG 导览问答

系统不是直接让大模型自由回答，而是先检索本地景区资料，再让大模型基于资料生成回答。后台可以展示每个回答命中的知识来源，便于评审看到准确率保障机制。

### 8.2 游客画像驱动的差异化讲解

同一个景点可以根据游客兴趣生成不同讲解版本。历史爱好者听到文化脉络，亲子游客听到故事化讲解，时间紧张游客听到精简版亮点。

### 8.3 情绪感知导览服务

系统识别游客焦虑、抱怨、疲惫等情绪后，调整数字人语气和推荐策略，使导览服务不只是回答问题，还能处理真实旅游场景中的体验波动。

### 8.4 游客反馈到运营建议的闭环

系统将问答日志、满意度、停留时长和消费数据转化为运营洞察，帮助景区发现高频需求、内容缺口和服务痛点。

### 8.5 弱定位场景补偿

在 GPS 不稳定或室内定位困难时，系统允许游客手动选择当前景点，也可以通过“附近景点列表”“二维码点位”“景点图片识别”补偿定位不足。比赛阶段优先实现手动选择和二维码点位方案。

## 10. 测试与验收方案

### 10.1 功能测试

游客端测试：

1. 文本问答是否可用。
2. 语音问答是否可用。
3. 数字人是否能正常播报。
4. 字幕是否同步。
5. 表情和口型是否随语音变化。
6. 路线推荐是否根据兴趣和时长变化。
7. 游客反馈是否记录。

管理端测试：

1. 知识文档是否可上传。
2. 知识片段是否可检索。
3. 数字人配置是否生效。
4. 问答日志是否完整记录。
5. 数据大屏是否正确展示。
6. 感受度报告是否能生成。

### 10.2 准确率测试

建立不少于 100 条景区事实问答测试集，覆盖：

1. 景点位置。
2. 景点特色。
3. 历史文化。
4. 演出时间。
5. 游览建议。
6. 路线推荐。
7. 模糊问法。
8. 干扰性问题。

验收标准：

1. 事实性问答准确率不低于 90%。
2. 对知识库未覆盖问题不编造具体事实。
3. 同义问法能够命中正确知识片段。
4. 回答语言自然，适合导游场景。

### 10.3 性能测试

核心指标：

1. 文本问答响应时间控制在 3 秒以内。
2. 语音问答整体响应时间控制在 5 秒左右。
3. 知识检索耗时控制在 500 毫秒以内。
4. 数据大屏查询常用指标控制在 2 秒以内。
5. 连续演示 10 分钟不崩溃、不明显卡顿。

### 10.4 演示验收

最终演示必须覆盖以下脚本：

1. 游客选择兴趣和游览时长。
2. 游客语音询问灵山大佛相关问题。
3. 数字人完成语音、字幕、口型和表情回答。
4. 游客请求个性化路线推荐。
5. 游客表达不满或疲惫，系统进行情绪化服务。
6. 管理员查看问答日志和热门问题。
7. 管理员查看数据大屏和游客感受度报告。
8. 管理员上传或更新知识库并测试问答。

## 11. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 大模型回答幻觉 | 影响准确率 | 使用 RAG、证据约束、无答案拒答、测试集评测 |
| 语音链路延迟过高 | 影响体验 | 文本先行展示、流式 TTS、缓存热门问题、控制回答长度 |
| 数字人效果不稳定 | 影响演示 | 优先 2D 稳定方案，再做增强效果 |
| 知识库资料混乱 | 影响检索质量 | 按景点和字段结构化切分，建立标签和版本 |
| 后台数据不够真实 | 影响实用价值 | 使用官方 Excel 行为数据构建运营分析 |
| 功能太多做不完 | 影响完整度 | 优先完成游客问答、数字人播报、知识库、路线推荐、大屏核心闭环 |
| 演示现场网络波动 | 影响模型调用 | 准备本地缓存问答、离线知识库和预置演示数据 |

答辩时建议用一句话概括项目：

“灵境导游不是一个简单的景区聊天机器人，而是一个以 AI 数字人为入口、以本地知识库保障准确性、以游客行为数据驱动运营优化的智慧景区导览系统。”

重点表达：

1. 面向真实景区痛点：导游不足、信息单向、反馈缺失、服务不可量化。
2. 技术链路完整：ASR + RAG + 大模型 + TTS + 口型同步 + 情感分析。
3. 准确率可验证：本地知识库、检索证据、测试集评估。
4. 体验有特色：个性化讲解、情绪安抚、路线推荐、数字人沉浸表达。
5. 运营有价值：问答日志和行为数据转化为管理建议。

## 16. 项目成功标准

项目完成后应达到以下标准：

1. 功能链路完整，游客端和管理后台都可以稳定演示。
2. 景区事实问答准确率达到或超过 90%。
3. 语音问答响应延迟接近或低于 5 秒。
4. 数字人语音、字幕、口型和表情具有一致性。
5. 个性化路线推荐能体现不同游客画像差异。
6. 管理后台能展示真实数据分析价值。
7. 提交材料完整，PPT、文档和视频能清晰呈现项目亮点。

## 17. 结论

本项目以“AI 数字人导游”为核心产品形态，围绕游客体验和景区运营两个方向构建完整系统。游客端解决“问得到、听得懂、走得顺、体验好”的问题；管理端解决“看得见、管得动、能优化、可决策”的问题。

相比单纯的语音助手或普通导览系统，本项目通过本地知识库、RAG 问答、多模态交互、个性化路线、情感分析和数据大屏形成闭环，能够充分覆盖 A5 赛题的功能要求、技术要求和评分重点。只要按计划推进，优先保证核心闭环稳定，再逐步增强数字人表现力和数据分析深度，就能形成一套完成度高、演示效果强、文档表达清晰的参赛作品。
